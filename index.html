Contexto

O que é Edge Computing

MODELO TRADICIONAL ─────────────────────────────── [Usuário SP 🇧🇷] │ │ ~180ms (round-trip) │ [Servidor único — Virginia 🇺🇸] │ [Banco de dados] ▸ Toda requisição cruza o oceano. ▸ Um ponto de falha para todos.

EDGE COMPUTING ─────────────────────────────── [Usuário SP 🇧🇷] ── [Edge Brasil ] ─┐ [Usuário DE 🇩🇪] ── [Edge Alemanha] ─┼─ [Origin] [Usuário JP 🇯🇵] ── [Edge Japão ] ─┘ ▸ Código roda PERTO do usuário. ▸ ~10ms de latência. ▸ Sem servidor dedicado.
  • ⚡
    Latência mínima
    Código executa no nó mais próximo — 10–30ms vs 150–300ms no modelo centralizado.
  • 🌍
    Escala global automática
    Sem configurar instâncias por região. A rede distribui automaticamente em centenas de PoPs.
  • 💰
    Pay-per-request
    Cloudflare Workers cobra por 100k requisições — não por hora de VM idle.
  • 🔒
    Alta resiliência
    Sem single point of failure. Uma região sai do ar, as outras assumem.
  • 🚀
    Cold start negligenciável
    Workers isolados por V8 Isolates inicializam em microssegundos — não em segundos como Lambda.
O problema

Modelo tradicional: as dores reais

🐌 Latência geográfica

Servidor em Virginia, usuário no Brasil → 180ms só de round-trip.

Cada 100ms a mais reduz conversão em até 7%.

📍 Single region

Tudo em uma região. Se cair, tudo cai.

Escalar geograficamente = replicar infra × N regiões.

❄️ Cold starts pesados

AWS Lambda com Node.js: 300ms–1s+ no primeiro hit após inatividade.

Usuário sente e abandona.

💸 Custo de escala vertical

VMs rodam 24/7 mesmo com tráfego zero. Escalabilidade horizontal = mais servidores + LB + auto-scaling groups = mais custo + mais complexidade.

⚙️ Complexidade operacional

Load balancer, CDN, health checks, auto-scaling, deploys por região… tudo isso só para manter um servidor disponível globalmente.

💡 Edge Computing resolve tudo isso executando seu código na rede de distribuição global — perto do usuário, sem servidor para gerenciar.

Express — parte 1 de 2

CRUD com Express — GET + POST

Express (Node.js)
// server.js
const express = require('express')
const app     = express()

// Precisa declarar middleware para ler JSON
app.use(express.json())

let users  = [
  { id: 1, name: 'Alice', email: 'alice@example.com' },
  { id: 2, name: 'Bob',   email: 'bob@example.com'   },
]
let nextId = 3

// GET /users
app.get('/users', (req, res) => {
  res.json(users)        // res.json() funciona bem
})

// POST /users
app.post('/users', (req, res) => {
  const { name, email } = req.body  // body já parseado
  if (!name || !email) {
    return res.status(400).json({
      error: 'name and email are required'
    })
  }
  const user = { id: nextId++, name, email }
  users.push(user)
  res.status(201).json(user)
})

app.listen(3000) // ❌ Não roda em edge
Hono (Edge / Multi-runtime)
// src/index.ts
import { Hono } from 'hono'

const app = new Hono()

// Sem setup extra — Hono já lê JSON nativamente

let users = [
  { id: 1, name: 'Alice', email: 'alice@example.com' },
  { id: 2, name: 'Bob',   email: 'bob@example.com'   },
]
let nextId = 3

// GET /users
app.get('/users', (c) => {
  return c.json(users)  // ✅ API idêntica ao Express
})

// POST /users
app.post('/users', async (c) => {
  const { name, email } = await c.req.json()
  if (!name || !email) {
    return c.json(
      { error: 'name and email are required' }, 400
    )
  }
  const user = { id: nextId++, name, email }
  users.push(user)
  return c.json(user, 201) // ✅ status como 2º arg
})

export default app // ✅ Roda em Workers, Bun, Deno…

CRUD — PUT + DELETE

Express (Node.js)
// PUT /users/:id
app.put('/users/:id', (req, res) => {
  const id  = parseInt(req.params.id) // sem tipagem
  const idx = users.findIndex((u) => u.id === id)

  if (idx === -1) {
    return res.status(404).json({ error: 'Not found' })
  }

  users[idx] = { ...users[idx], ...req.body }
  res.json(users[idx])
})

// DELETE /users/:id
app.delete('/users/:id', (req, res) => {
  const id  = parseInt(req.params.id)
  const idx = users.findIndex((u) => u.id === id)

  if (idx === -1) {
    return res.status(404).json({ error: 'Not found' })
  }

  users.splice(idx, 1)
  res.status(204).send()
})
Hono (Edge / Multi-runtime)
// PUT /users/:id
app.put('/users/:id', async (c) => {
  const id  = Number(c.req.param('id')) // ✅ tipado
  const idx = users.findIndex((u) => u.id === id)

  if (idx === -1) {
    return c.json({ error: 'Not found' }, 404)
  }

  const body = await c.req.json()
  users[idx] = { ...users[idx], ...body }
  return c.json(users[idx])
})

// DELETE /users/:id

  if (idx === -1) {
    return c.json({ error: 'Not found' }, 404)

  return new Response(null, { status: 204 })

Conceito fundamental

O que é Middleware?

💡 Analogia simples

Imagine o porteiro de um prédio corporativo.
Toda visita (requisição) que chega, passa pelo porteiro primeiro — ele verifica o crachá, anota no livro e só então deixa entrar.

Middleware é exatamente isso: código que roda antes (e/ou depois) do seu handler, sem que o handler precise saber que ele existe.

Fluxo de uma requisição
📱 Cliente
──── requisição chega ────▶
🔒 Middleware: Auth — tem token? se não: 401 Unauthorized
✓ token válido → segue
📝 Middleware: Logger — registra a requisição no log
✓ registrado → segue
🎯 Route Handler — finalmente executa a lógica
◀──── resposta volta ────
📱 Cliente
Como escrever um Middleware no Hono
// Um middleware é uma função com 2 argumentos:
// c = contexto da requisição
// next = função que chama o próximo passo

const authMiddleware = async (c, next) => {
  const token = c.req.header('Authorization')

  if (!token) {
    // ❌ Sem token: para aqui, devolve 401
    return c.json({ error: 'Sem autorização' }, 401)
  }

  // ✅ Com token: chama o próximo (handler ou middleware)
  await next()
}

// Aplicar apenas nas rotas /admin/*
app.use('/admin/*', authMiddleware)

app.get('/admin/users', (c) => {
  // Só chega aqui se o token for válido!
  return c.json({ users: [] })
})
  • 🔑
    Auth — bloqueia quem não tem permissão
  • 📝
    Logger — anota cada requisição que chegou
  • 🌐
    CORS — libera (ou bloqueia) origens externas
  • 🛡️
    Secure Headers — adiciona cabeçalhos de segurança automaticamente
Arquitetura

Estrutura recomendada de projeto

Organização de pastas
src/
├── index.ts           ← Liga tudo: registra rotas
│
├── routes/            ← Rotas HTTP (GET, POST…)
│   ├── users.ts
│   └── auth.ts
│
├── services/          ← Lógica de negócio
│   └── userService.ts
│
├── middleware/        ← Middlewares customizados
│   └── authMiddleware.ts
│
└── types/             ← Tipos TypeScript
    └── index.ts
💡 Cada pasta tem uma responsabilidade só.
Rotas não sabem do banco. Services não sabem do HTTP.
Middleware não sabe da lógica de negócio.
Camadas da aplicação
🌐 Routes — recebe HTTP, chama o Service, devolve JSON
▼ chama
⚙️ Services — regras de negócio, validações, lógica
▼ acessa
🗄️ Data Layer — banco de dados, KV Store, APIs externas

Boas práticas

Usando Hono do jeito certo

// ✅ Middleware global — roda em todas as rotas
app.use('*', cors())
app.use('*', logger())
app.use('*', secureHeaders())

// ✅ Middleware por rota — só onde precisa
app.use('/admin/*', bearerAuth({ token: env.TOKEN }))

// ✅ Validação com Zod — nunca confie nos dados!
import { zValidator } from '@hono/zod-validator'
import { z }          from 'zod'

const userSchema = z.object({
  name:  z.string().min(1),
  email: z.string().email(),
})

app.post('/users', zValidator('json', userSchema), async (c) => {
  const body = c.req.valid('json') // 100% tipado ✅
  // body.name e body.email já validados
})

// ✅ Error handler global — não deixe erros vazarem
app.onError((err, c) => {
  return c.json({ error: err.message }, 500)
})

// ✅ Not Found handler centralizado
app.notFound((c) => {
  return c.json({ error: 'Route not found' }, 404)
})
  • ✅
    Valide na entrada com Zod
    Use @hono/zod-validator para validar body, params e query antes do handler. Jamais confie nos dados do cliente.
  • ✅
    Middleware separado por responsabilidade
    Global para CORS/logging, específico para autenticação. Não misture preocupações no mesmo middleware.
  • ✅
    Um arquivo de rotas por domínio
    users.ts, products.ts, orders.ts. O index.ts só monta — não implementa.
  • ⚠️
    Não reinvente o middleware
    CORS, JWT, Logger, ETag já existem no Hono. Só crie middleware custom quando houver necessidade real.
  • ⚠️
    Evite overengineering cedo
    Para projeto simples, um arquivo único está OK. Modularize quando o time e a complexidade exigirem de verdade.
Decisão técnica

Quando usar e quando evitar

✅ Use Hono quando...

  • 🚀
    Você quer uma API REST ou RPC leve rodando no Edge (Cloudflare Workers, Bun, Deno)
  • 🔌
    Construindo um BFF (Backend for Frontend) — camada fina que agrega dados de múltiplos serviços
  • 🧩
    Proxy ou gateway na frente de uma CDN ou de uma API monolítica
  • 📦
    Projeto com bundle crítico — Workers têm limite de 1MB no Cloudflare
  • 🔑
    Time que usa TypeScript e quer DX moderna sem abrir mão de performance
  • 🌐
    Full-stack com JSX — Hono suporta SSR com componentes JSX no servidor

❌ Evite Hono quando...

  • 💾
    Sua aplicação usa ORMs com drivers nativos como Prisma — incompatíveis com runtimes de edge
  • 🧠
    Você precisa de estado em memória persistente entre requisições — Workers são stateless por design
  • 📁
    Upload/download de arquivos grandes — Workers têm timeout de 30s e memória limitada (~128MB)
  • 🐘
    Time já usa NestJS ou Express consolidado com ecossistema de plugins e decoradores próprios
  • ⏱️
    Processamento CPU-intensivo ou de longa duração — edge não é o ambiente certo para isso

💡 Combine: Hono no Edge para a camada de API + serviço dedicado (Node.js, Go) para processamento interno pesado.

📖 Glossário de Termos Técnicos
Passe o mouse sobre palavras com sublinhado pontilhado nos slides para ver a definição rápida. Ou consulte todos os termos aqui — ESC ou clique fora para fechar.
Edge Computing
Executar código em servidores distribuídos globalmente, próximos ao usuário, ao invés de em um datacenter central único. Reduz latência drasticamente.
PoP (Point of Presence)
Ponto de Presença: servidor ou datacenter distribuído geograficamente. A Cloudflare tem mais de 310 PoPs ao redor do mundo.
CDN
Content Delivery Network: rede de servidores espalhados pelo mundo para entregar conteúdo de forma mais rápida, próxima ao usuário final.
Cold Start
Tempo de inicialização de uma função serverless inativa. No AWS Lambda pode ser 300ms–1s+. No Edge com V8 Isolates é quase zero.
V8 Isolates
Contextos de execução isolados do motor JavaScript V8 (o mesmo do Chrome). Muito mais leves que containers — inicializam em microssegundos.
Single Point of Failure
Ponto único de falha: componente que, se cair, derruba todo o sistema. Em arquiteturas distribuídas, isso é eliminado com redundância.
VM (Virtual Machine)
Máquina Virtual: servidor virtual que emula um computador completo. Fica rodando 24h/dia, gerando custo mesmo sem receber requisições.
Round-trip (RTT)
Tempo de ida e volta de uma requisição — do cliente ao servidor e de volta. Quanto maior a distância geográfica, maior o RTT.
Runtime
Ambiente de execução de um programa. Ex: Node.js é um runtime para JavaScript. Cada runtime tem suas próprias APIs e limitações.
Bundle
Arquivo único gerado ao empacotar o código-fonte para distribuição. Bundle menor = menos dados = carregamento mais rápido.
RegExpRouter
Roteador do Hono baseado em expressões regulares. Compila todas as rotas em uma única Regexp antes do primeiro dispatch — extremamente rápido.
Middleware
Código executado entre a requisição e a resposta. Ex: verificar autenticação antes de processar uma rota protegida.
CORS
Cross-Origin Resource Sharing: mecanismo de segurança dos navegadores que controla quais domínios podem acessar sua API.
JWT
JSON Web Token: token de autenticação com informações do usuário codificadas e assinadas digitalmente — sem precisar de sessão no servidor.
ETag
Entity Tag: identificador de versão de um recurso HTTP. Permite que o navegador saiba se o conteúdo mudou sem precisar baixá-lo novamente.
DX (Developer Experience)
Experiência do Desenvolvedor ao usar uma ferramenta. Boa DX = menos código repetitivo, erros detectados no editor e APIs intuitivas.
BFF (Backend for Frontend)
Camada de backend criada especificamente para servir um frontend. Agrega dados de múltiplos serviços internos em um único endpoint otimizado para a tela.
SSR (Server-Side Rendering)
Renderização no Servidor: o HTML da página é gerado no servidor antes de ser enviado ao navegador. Melhora performance inicial e SEO.
ORM
Object-Relational Mapping: biblioteca que mapeia objetos do código a tabelas do banco de dados. Ex: Prisma, TypeORM. Muitos não funcionam em edge.
Stateless
Sem estado: cada requisição é completamente independente. O servidor não guarda nenhuma informação entre chamadas — base da escalabilidade do edge.
RPC
Remote Procedure Call: o cliente chama funções no servidor como se fossem locais, com tipagem compartilhada. O Hono suporta modo RPC com Zod.
Pay-per-request
Modelo de cobrança onde você paga apenas pelas requisições feitas, não pelo tempo que o servidor fica ligado (diferente de VMs que cobram por hora).
Boilerplate
Código repetitivo e previsível que você escreve sempre igual, sem agregar valor à lógica do sistema. Ex: montar o header "Content-Type: application/json" em toda rota. Frameworks existem para eliminar boilerplate.
